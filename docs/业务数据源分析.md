# 业务数据源分析

## 1. 概述

本分析基于 `docs/raw/` 目录中的原始数据，梳理 ForgeFin 财务录入凭证所需的基础数据来源、数据流转路径、可用字段质量以及系统对接建议。分析范围限定于已提供的样本数据，忽略不完整或无法验证的字段。

## 2. 数据源清单

| 数据源 | 文件名 | 来源 | 更新频率 | 关键字段 | 用途 |
|--------|--------|------|----------|----------|------|
| 银行流水 | `银行流水.tsv` | 对应银行，第二个工作日导出 | 每日/次日 | 交易时间、对方单位、转入/转出金额、余额、摘要、附言 | 现金日记账、银行对账、凭证基础 |
| POS 订单流水 | `pos流水.tsv` | 工商银行商户网站导出 | 按需导出 | 交易时间、协议名称、工行订单号、订单金额、订单状态、结算状态 | 交易明细核对、汇总生成 |
| 订单流水 | `订单流水.tsv` | 工商银行商户网站导出 | 按需导出 | 交易时间、工行订单号、订单金额、手续费、商户实收金额、支付方式、借贷记标识、结算日期 | 手续费拆分、实收金额核对、收入确认 |
| 数据汇总 | `数据汇总.tsv` | 人工根据上述数据整理，备注来自微信群 | 每日/按需 | 日期、收据编号、一级分类、项目、事由、收入、支出、余额、备注 | 凭证生成的直接依据 |

## 4. 各数据源详细分析

### 4.1 银行流水

**来源说明：** 银行流水在交易日后的第二个工作日由银行生成并提供，反应公司银行账户的实际资金变动。

**字段可用性分析：**

| 字段 | 完整性 | 说明 |
|------|--------|------|
| 凭证号 | 低 | 示例全部为 `000000000`，无实际凭证号，不可用 |
| 对方账号 | 中 | 部分有账号，部分为空 |
| 交易时间 | 高 | 精确到秒，格式统一 |
| 对方单位 | 中 | 转入方/转出方名称，部分为空（如手续费、POS 清算） |
| 用途 | 低 | 多数为空 |
| 摘要 | 高 | 交易类型描述，如“货款”、“报销”、“跨行汇款手续费” |
| 附言 | 中 | POS 清算流水的附言为商户编号 `100172984575` |
| 转入金额 | 高 | 收入金额 |
| 转出金额 | 高 | 支出金额 |
| 余额 | 高 | 交易后账户余额 |

**主要用途：**
- 银行日记账自动生成
- 银行余额核对
- 识别收入到账与支出付款

**局限性：**
- POS 清算入账为合并金额（T+1 结算，单日多笔合并），无法直接对应单笔交易
- 缺少客户/事由信息，需要配合订单流水和人工汇总

### 4.2 POS 订单流水

**来源说明：** 从银行工商商户网站导出，网站登录存在防机器人机制，暂不支持自动抓取。

**字段可用性分析：**

| 字段 | 完整性 | 说明 |
|------|--------|------|
| 交易时间 | 高 | 精确到秒 |
| 协议编号/名称 | 高 | 固定为 `0301锦绣健康管理有限公司` |
| 工行订单号 | 中 | 显示为脱敏格式，如 `1001...30135540` |
| 商户订单号 | 低 | 全部为 `-` |
| 订单金额 | 高 | 原始交易金额 |
| 累计退款金额 | 高 | 样本全部为 `0.00` |
| 订单状态 | 高 | 全部为 `已付款` |
| 结算状态 | 高 | 全部为 `已结算` |

**主要用途：**
- 按日汇总交易金额
- 与银行流水中的 POS 清算入账核对

**局限性：**
- 工行订单号脱敏，难以与订单流水完全精确匹配
- 缺少手续费、支付方式等关键字段，建议优先使用完整版 `订单流水.tsv`

### 4.3 订单流水（推荐主数据源）

**来源说明：** 与 `pos流水.tsv` 同源，但字段更完整，是生成数据汇总的主要依据。

**字段可用性分析：**

| 字段 | 完整性 | 说明 |
|------|--------|------|
| 商户名称 | 高 | `锦绣健康管理有限公司` |
| 商户编号 | 高 | `100172984575` |
| 产品名称 | 高 | `线下产品` |
| 协议名称/编号 | 高 | `锦绣健康管理有限公司` / `0301` |
| 工行订单号 | 高 | 完整订单号，如 `100172984575000052607130135540` |
| 订单金额 | 高 | 客户支付金额 |
| 手续费金额 | 高 | 精确到分 |
| 商户实收金额 | 高 | 订单金额 - 手续费 |
| 交易时间 | 高 | 精确到秒 |
| 支付方式 | 高 | `微信` / `支付宝` |
| 借贷记标识 | 高 | `借记卡` / `贷记卡` |
| 客户备注 | 低 | 仅部分有值 |
| 入账账号 | 高 | 脱敏账号 `1001****3418` |
| 结算状态/结算日期 | 高 | 可用于确认收入归属日期 |
| 第三方订单号 | 高 | 微信/支付宝流水号 |

**手续费率规则（来自 `数据汇总.tsv` 表头）：**

| 支付方式/卡种 | 费率 |
|--------------|------|
| 借记卡 | 0.5% |
| 信用卡（贷记卡） | 0.6% |
| 微信/支付宝 | 0.25% |

**费率验证：** 样本中微信支付订单的手续费率约为 `0.25%`（如 `12000.00` 对应手续费 `30.00`），与表头一致。支付宝订单的手续费也按 `0.25%` 计算。

**主要用途：**
- 自动确认营业收入
- 自动拆分手续费支出
- 与银行清算流水核对
- 生成客户维度收入明细

### 4.4 数据汇总

**来源说明：** 由财务人员根据银行流水、订单流水整理而成。其中的“事由”、“备注”信息需要从特定微信群获取。

**字段可用性分析：**

| 字段 | 完整性 | 说明 |
|------|--------|------|
| 日期 | 高 | 交易或记账日期 |
| 收据编号 | 中 | 部分收入有编号，支出多数为空 |
| 一级 | 高 | `收入` / `支出` |
| 项目 | 高 | 会计项目，如 `营业收入`、`手续费`、`产品成本` |
| 事由 | 高 | 业务描述 |
| 借记卡/信用卡/微信支付宝金额 | 中 | 仅收入类 POS 交易有值 |
| 手续费 | 中 | 对应 POS 交易手续费 |
| 实际收入 | 中 | 收入净额 |
| 支出 | 高 | 支出金额 |
| 余额 | 高 | 账户余额 |
| 备注 | 中 | 对方单位、清算账户等信息 |

**主要用途：**
- 直接作为凭证录入的草稿依据
- 明确每笔业务的会计科目和金额

**局限性：**
- 人工整理，存在录入错误风险
- 备注依赖微信群，数据来源不稳定，需要设计补充录入入口

## 6. 数据质量问题

1. **银行流水 POS 清算金额合并**：单日多笔 POS 交易合并为一笔银行转入，无法直接拆分，必须依赖订单流水
2. **银行流水中部分支出缺少对方单位**：如跨行汇款手续费未显示收款方，需要人工补录
3. **订单流水缺少客户/业务事由**：无法直接判断收入对应客户或项目，需要结合数据汇总
4. **数据汇总依赖微信群备注**：非结构化来源，无法自动导入，需设计人工补录机制
5. **POS 订单流水工行订单号脱敏**：完整订单号应以 `订单流水.tsv` 为准
6. **2026-07-01 清算金额存在 0.45 元差异**：需确认是否由补贴、立减或四舍五入导致
7. **数据汇总表头费率与实际不完全对应**：汇总表中有“借记卡手续费”“信用卡手续费”“微信/支付宝”三列，但实际仅“微信/支付宝”列有值，借记卡/信用卡交易通过微信/支付宝渠道完成，费率为 0.25%

## 7. 系统对接建议

### 7.1 数据导入

| 数据源 | 导入方式 | 频率 | 优先级 |
|--------|----------|------|--------|
| 银行流水 | 文件导入（银行网银导出） | 每日 | 高 |
| 订单流水 | 文件导入（工行商户网站导出） | 每日/按需 | 高 |
| 数据汇总 | 手工录入或从模板导入 | 每日 | 中 |
| 微信备注 | 提供补录字段，关联汇总记录 | 实时 | 中 |

### 7.2 凭证生成规则建议

- **POS 收入凭证**：
  - 借：银行存款（实收金额）
  - 借：财务费用-手续费（手续费金额）
  - 贷：主营业务收入（订单金额）Typora
- **普通支出凭证**：
  - 借：对应成本/费用科目
  - 贷：银行存款
- **手续费支出凭证**：
  - 借：财务费用-手续费
  - 贷：银行存款

### 7.3 核对机制建议

1. 每日自动比对订单流水实收金额合计与银行流水 POS 清算转入金额
2. 对差异记录生成待核对任务
3. 将数据汇总记录与银行流水、订单流水进行三方勾稽
4. 未匹配的银行流水和订单流水单独列示，提示人工处理

## 8. 结论

当前业务流程中：

- **银行流水**是资金实际变动的权威来源，但单笔 POS 交易已被合并清算，无法单独作为收入确认依据。
- **订单流水**是最完整的交易级数据来源，包含订单金额、手续费、实收金额、支付方式、结算日期，应作为收入确认和手续费拆分的主要依据。
- **数据汇总**是财务人员整理后的凭证草稿，其中的事由、备注来自微信群等非结构化来源，需要系统提供人工补录和审核机制。
- **POS 简化流水**字段过少，建议后续直接使用完整版订单流水，避免重复维护。

系统应优先实现银行流水导入、订单流水导入、自动核对与差异提示、以及基于核对结果的凭证生成功能。
