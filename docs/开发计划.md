# ForgeFin 开发计划

> 基于 README.md 和当前代码库分析

## 核心架构原则：多公司/多账套

- **数据隔离**：每个公司/账套使用独立的 SQLite `.db` 文件，业务数据完全互不干扰
- **系统库**：一个共享的 `forgefin_system.db` 管理用户信息和公司注册信息
- **公司库命名**：`forgefin_company_{company_id}.db`，按需创建和切换
- **切换机制**：用户登录后选择公司，所有业务操作自动路由到对应数据库文件
- **SQLite 策略**：保持精简，不引入 ORM 层，直接使用 rusqlite 操作。每个连接池管理一个数据库文件
- **备份策略**：每个公司的 `.db` 文件可独立备份和恢复。备份文件按日期/公司命名，支持手动立即备份和定期自动备份。恢复操作需二次确认

---

## 阶段任务规则

所有阶段的任务必须遵守以下规则：

| 规则 | 说明 |
|------|------|
| **多公司隔离** | 所有业务数据（科目、凭证、客户/供应商、报表等）必须按公司 ID 隔离，跨公司操作需明确设计 |
| **无 ORM** | 所有数据库操作使用 `rusqlite` 直接执行原始 SQL，严禁引入 ORM 框架 |
| **独立数据库文件** | 每公司一个独立 `.db` 文件，系统库与业务库分离。任务中涉及的数据存储必须遵循此结构 |
| **备份可恢复** | 任何数据变更操作都应考虑备份影响。重要功能（凭证保存/审核/删除等）需要确保数据可还原 |
| **Tauri 命令** | 前后端通信通过 Tauri `invoke` 命令完成，命令层需传递当前公司 ID |

## Phase 0：基础设施（Pre-MVP）

建立支撑所有业务模块的底层能力。

```
优先级: ★★★★★
```

| 任务 | 说明 |
|------|------|
| 0.1 | SQLite 数据层 —— 在 `src-tauri/src/db/` 建立数据库模块。使用 `rusqlite` 的 `Connection`，每公司一个数据库实例，用 `Mutex` 管理线程安全。不引入连接池，保持精简 |
| 0.2 | Tauri 命令层 —— 在 `src-tauri/src/commands/` 中建立命令模块，Tauri invoke 通信模式。所有命令需携带当前公司 ID，动态切换数据库实例 |
| 0.3 | 数据库 Schema —— 系统库（用户、公司/账套注册信息）+ 每个公司的独立业务库（会计科目表、凭证表、凭证分录表、客户/供应商表等） |
| 0.4 | 认证系统 —— 登录页 + 用户管理（用户可跨公司/账套，权限按公司独立控制） |
| 0.5 | 公司/账套管理 —— 多组织支持。每个公司独立数据库文件，业务数据完全隔离。支持新建/切换/编辑公司信息 |
| 0.6 | **备份与恢复** —— 支持手动/自动备份公司数据库文件，支持从备份文件恢复。备份粒度：单公司/全系统 |

## 迭代节奏

每 1-2 周完成 1 个小功能 → 立即在真实业务中使用 → 收集反馈 → 优化。

**最优先启动**：Phase 0 (数据层 + 认证) → Phase 1 (科目 + 凭证录入 + 凭证查询)。
